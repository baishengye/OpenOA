# 踩坑速查（实战记录）

按「现象 → 原因 → 解法」整理，均为本项目搭建三端时真实遇到并解决的问题。

---

## 通用 / pnpm monorepo

### 1. Metro 报 `Unable to resolve module invariant from react-native/...Button.js`
- **原因**：`metro.config.js` 里设了 `resolver.disableHierarchicalLookup = true`，破坏了 pnpm `.pnpm` 嵌套 node_modules 的逐级向上查找，react-native 的传递依赖（invariant 等）找不到。
- **解法**：**移除** `disableHierarchicalLookup`。pnpm 下必须保留逐级查找。见 [apps/oa/metro.config.js](../apps/oa/metro.config.js)。

### 2. pnpm + RN 各种解析/autolink/codegen 失败
- **原因**：pnpm 默认符号链接结构，RN 假设扁平 node_modules。
- **解法**：[.npmrc](../.npmrc) 设 `node-linker=hoisted` + `shamefully-hoist=true`。

### 3. builder-bob 构建报 `exports['.'].types should not be set when esm enabled` 或消费方把 enum 当 `{}`
- **原因**：esm 模式下 `package.json` 的 `exports` 若把 `default` 直接指向 `.js`，TS 在其旁找不到 `.d.ts`（类型在 `lib/typescript/module/`）。
- **解法**：`exports['.']` 用 `import: { types, default }` 嵌套，`types` 指向 `./lib/typescript/module/index.d.ts`。见各包 package.json。

### 4. 库 tsc 报 `TS6305 / TS6059`（输出文件未构建 / 不在 rootDir）
- **原因**：给库 tsconfig 加了 `composite`+`references` 或把 `paths` 指向 base **源码**，与 builder-bob 输出布局冲突 / 把 base 源码拉进 rootDir。
- **解法**：库 tsconfig **不要** composite/references/paths-to-source；靠 pnpm workspace 从 node_modules 解析 base 的**已构建**类型（`pnpm build` 拓扑序保证 base 先构建）。

### 5. `ItcError` 报 `override` 缺失（TS4114）
- **原因**：`target: esnext` 下 `Error` 已有 `cause`，覆盖需显式 `override`。
- **解法**：`override readonly cause?`。见 [packages/base/src/result.ts](../packages/base/src/result.ts)。

## Android

### 6. autolinking 报 `Could not find project.android.packageName in react-native config output`
- **原因**：RN 0.77 把 CLI 拆出，`react-native config` 依赖 `@react-native-community/cli`，未安装则 autolink 拿不到配置。
- **解法**：把 `@react-native-community/cli`、`-platform-android`、`-platform-ios`（15.0.1）加为 app devDependency。

### 7. settings.gradle `includeBuild('../node_modules/@react-native/gradle-plugin')` 找不到
- **原因**：`@react-native/gradle-plugin`、`@react-native/codegen` 是 react-native 的**传递**依赖，pnpm 不会软链进 `apps/oa/node_modules`。
- **解法**：把这两个包加为 app 的**直接** devDependency，pnpm 即软链到位，模板默认相对路径生效。

### 8. Gradle 找不到 Java / `Unable to locate a Java Runtime`
- **原因**：未装独立 JDK。
- **解法**：用 Android Studio 自带 JBR：`export JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home"`（JDK 21）。

### 9. compileSdk 35 / NDK 27.1 未安装
- **解法**：对齐本机已装版本 —— [android/build.gradle](../apps/oa/android/build.gradle) 设 `compileSdkVersion=36`、`ndkVersion=26.1.10909125`。

### 10. 库模块 buildscript 插件 classpath 冲突
- **原因**：feature 库 `android/build.gradle` 自带 `com.android.tools.build:gradle` classpath，与宿主根工程的 AGP 版本冲突。
- **解法**：库只声明 kotlin 插件 classpath（版本取自 `rootProject.ext.kotlinVersion`），AGP 由宿主提供。见 [packages/biometric/android/build.gradle](../packages/biometric/android/build.gradle)。

### 11. guava / maven 依赖 `Remote host terminated the handshake`
- **原因**：网络抖动。**解法**：重试 `./gradlew` 即可。

### 11b. 接 `react-native-code-push@8.3.1` 到 RN 0.82（AGP 8.12）的 4 连坑
老库（微软 App Center 退役、长期不更新）接新 RN，依次会撞 4 个，全部修完才能编过：
1. **`strings.xml` 缺 `CodePushDeploymentKey` → 编译失败**。主包 `react-native.config.js` 的 `packageInstance` 用 `getResources().getString(R.string.CodePushDeploymentKey)`，autolink 生成的 PackageList 引用它。**解法**：[strings.xml](../apps/oa/android/app/src/main/res/values/strings.xml) 加 `<string name="CodePushDeploymentKey"></string>`（空也必须有）。
2. **`Could not resolve project :react-native-code-push / No variants exist`**。主包的 `android/` 是**根工程级** build.gradle（AGP 1.3.0 classpath），真正的 library 在 `android/app/`。**解法**：[react-native.config.js](../apps/oa/react-native.config.js) 给它配 `dependencies['react-native-code-push'].platforms.android.sourceDir = '../node_modules/react-native-code-push/android/app'`。
3. **`Namespace not specified`**。该 library 用旧式 `AndroidManifest package=`，无 `namespace`，AGP 8 强制要。**解法**：[android/build.gradle](../apps/oa/android/build.gradle) 用 `subprojects { … plugins.withId("com.android.library") { android { if (namespace == null) namespace "com.microsoft.codepush.react" } } }` 注入（**不能用 `afterEvaluate`**——该子项目可能已被 RN 插件提前求值，报 `Cannot run afterEvaluate when already evaluated`）。
4. **`程序包 ChoreographerCompat 不存在`**。`CodePushNativeModule.java` 用了 RN 0.74+ 已删的 `com.facebook.react.modules.core.ChoreographerCompat`。**解法**：patch 成 `android.view.Choreographer`（`ChoreographerCompat.FrameCallback` → `Choreographer.FrameCallback`，`doFrame(long)` 签名一致）。用 **pnpm patch 持久化**（`pnpm patch react-native-code-push@8.3.1 --edit-dir … && pnpm patch-commit …`）→ 生成 [patches/react-native-code-push@8.3.1.patch](../patches/react-native-code-push@8.3.1.patch)，pnpm 11 把配置写进 `pnpm-workspace.yaml` 的 `patchedDependencies`，`pnpm install` 自动重打。
> iOS 侧无这些坑：CodePush 是老架构 `NativeModule`，主包自带现成 ObjC，autolink + `pod install` 直接集成（前提是主包 `react-native-code-push` 在 `apps/oa/package.json` dependencies，否则 autolink 扫不到 → `NativeModules.CodePush` undefined → `ItcHotfix` 顶层读 `InstallMode` 崩）。详见 [鸿蒙接入要点.md §8](鸿蒙接入要点.md)。

## iOS

### 12. `bundle install` 报 `ffi-1.17 requires ruby >= 3.0`（系统 Ruby 2.6.10）
- **解法**：[Gemfile](../apps/oa/Gemfile) 钉住 `ffi < 1.17`、`securerandom < 0.4`、`logger < 1.6`。

### 13. `Bundler::SudoNotPermittedError`（想装到系统目录）
- **原因**：`bundle config set --local path` 未生效。
- **解法**：用环境变量：`BUNDLE_PATH=vendor/bundle BUNDLE_DISABLE_SHARED_GEMS=true bundle install`，把 gem 装到项目本地 `vendor/bundle`。

### 14. `xcodebuild ... Unable to find a destination ... iOS 26.5 is not installed`
- **原因**：Xcode 只有 SDK 头文件，未下载 iOS 平台/模拟器运行时。
- **解法**：`xcodebuild -downloadPlatform iOS`（约 7GB，装到系统盘）。

### 15. 编译 `fmt/format-inl.h: call to consteval function ... is not a constant expression`
- **原因**：Xcode 26.5 新版 Clang 下，fmt **11.0.0.2** 的 `base.h` 会**无条件**把 `FMT_USE_CONSTEVAL` 重定义为 1（无 `#ifndef` 守卫），导致 `-DFMT_USE_CONSTEVAL=0` 宏**无效**；而该 clang 的 consteval 对 fmt 用法过严，编译失败。
- **解法**：直接 **patch `Pods/fmt/include/fmt/base.h`**，在 `#if FMT_USE_CONSTEVAL` 块前插入 `#undef FMT_USE_CONSTEVAL` + `#define FMT_USE_CONSTEVAL 0` 强制关闭。已写进 [Podfile](../apps/oa/ios/Podfile) 的 `post_install`（每次 `pod install` 自动重打，幂等）。
  > 注意：单纯给 `GCC_PREPROCESSOR_DEFINITIONS` 加 `FMT_USE_CONSTEVAL=0` 对 fmt 11.0 无效——头文件会覆盖它。必须改头文件。

### 16. Apple Silicon 上编译模拟器在 x86_64 切片失败
- **解法**：加 `ONLY_ACTIVE_ARCH=YES`（只编 arm64）。Release 默认编全架构。

### 16b. Xcode 跑 Debug 报 `tar: hermes-ios-0.77.0-debug.tar.gz: No such file or directory`
- **现象**：Release 能编、**Debug 失败**（Xcode GUI 默认 Debug）。
- **原因**：`hermes-engine` pod 只下载了 release 预编译包，**debug 包没下下来**（`Pods/hermes-engine-artifacts/` 里只有 `hermes-ios-0.77.0-release.tar.gz`）。
- **解法**：手动补下 debug 包到该目录：
  ```bash
  cd apps/oa/ios/Pods/hermes-engine-artifacts
  curl -fL -o hermes-ios-0.77.0-debug.tar.gz \
    https://repo1.maven.org/maven2/com/facebook/react/react-native-artifacts/0.77.0/react-native-artifacts-0.77.0-hermes-ios-debug.tar.gz
  ```
  （URL 里的 `0.77.0` 换成你的 RN 版本。）

### 17. 真机 .ipa 无法产出
- **原因**：Apple 要求开发者签名（Team + Provisioning + 设备 UDID），无法代办。
- **解法**：Xcode 打开 `.xcworkspace` 选自己的 Team → Archive → Distribute。

## 鸿蒙 (HarmonyOS / RNOH)

### 19. Metro dev server 跑鸿蒙白屏：`Unable to resolve module .../ReactDevToolsSettingsManager`
- **现象**：`pnpm start:harmony` 后打开 App 白屏，Metro 报 `Unable to resolve module ../../src/private/devsupport/rndevtools/ReactDevToolsSettingsManager from .../@react-native-oh/react-native-harmony/Libraries/Core/setUpReactDevTools.js`。
- **原因**：RNOH 0.82.30 的 `ReactDevToolsSettingsManager` **只有 `.android.js`/`.ios.js`**，缺 harmony/通用变体。它被 `setUpReactDevTools.js` 在 `if (__DEV__)` 块里 require：
  - **离线 release 包**（`__DEV__=false`，DevEco 跑的 `bundle.harmony.js`）—— 死代码消除删掉整块，**从不 require**，所以离线包一直正常；
  - **Metro dev server**（`__DEV__=true`）—— 首次真正执行 dev-only 代码才解析它，harmony 平台找不到文件 → 白屏。
  - 所以**与业务改动（如 code_push）无关**，是首次用 dev server 跑鸿蒙才暴露的 RNOH 问题。
- **解法**：在 [metro.config.harmony.js](../apps/oa/metro.config.harmony.js) 用 `resolver.resolveRequest` 仅对 `platform==='harmony'` 把该模块重定向到本地 stub [src/stubs/ReactDevToolsSettingsManager.js](../apps/oa/src/stubs/ReactDevToolsSettingsManager.js)（`getGlobalHookSettings` 返回 null、`setGlobalHookSettings` no-op）。匹配用**精确尾路径** `devsupport/rndevtools/ReactDevToolsSettingsManager`，避免 `endsWith('ReactDevToolsSettingsManager')` 误伤 `specs/Native…SettingsManager`。改 config 后需 `--reset-cache` 重启 Metro。
  > ⚠️ 加这个自定义 `resolveRequest` 时**必须委托回 RNOH 的原始实现**，否则会覆盖它、连锁触发 **#21**（Platform/三方包重定向全失效）。务必照 #21 的 `delegateResolve` 写法。
  > 该包其余 dev 文件（`setUpFuseboxReactDevToolsDispatcher`、两个 specs）都有通用 `.js`，只此一个缺。

### 20. 鸿蒙模拟器/真机连不上 Metro：`No apps connected ... Sending "reload" failed`
- **原因**：鸿蒙没有 `adb reverse`，设备访问本机 Metro(8081) 需 `hdc rport tcp:8081 tcp:8081`。直接 `react-native start` 不会自动设端口转发。
- **解法**：用 `pnpm start:harmony`（其 [scripts/harmony-rport.mjs](../apps/oa/scripts/harmony-rport.mjs) 会先设端口转发再启 Metro）；或手动 `hdc rport tcp:8081 tcp:8081` 后 reload。

### 21.（重要）自定义 metro `resolveRequest` 覆盖了 RNOH 的 → 三方包/Platform 重定向全失效
- **现象**（依触发顺序，可能逐个出现）：
  - `TurboModuleProvider.cpp> Couldn't provide turbo module "CodePush"` + JS `Cannot read property 'InstallMode' of undefined`；
  - 修了上面后又变 `TypeError: Cannot read property 'select' of undefined`（`Platform.select`，且崩在 RN 核心组件如 Button/ScrollView 的样式里）。
- **根因**：`createHarmonyMetroConfig` **自带** `resolver.resolveRequest`，它在 `platform==='harmony'` 时自动做两件事：① 把 `react-native` → `@react-native-oh/react-native-harmony`（提供 `Platform`/`AppState` 等 harmony 实现）；② 把三方库主包 → oh-tpl 移植包（`react-native-code-push`/`react-native-mmkv-storage`/`@op-engineering/op-sqlite`，见 `bundle-harmony` 日志 `Redirected imports to N ... package(s)`）。
  若你为了别的目的（如 #19 加 stub）在 [metro.config.harmony.js](../apps/oa/metro.config.harmony.js) 自定义 `resolver.resolveRequest`，`mergeConfig` 对同名函数字段是**后者覆盖**，于是 RNOH 那套自动重定向**全部失效**：code-push 解析到主包（`NativeModules.CodePush` 鸿蒙没有）、`Platform` 解析到标准 RN（无 harmony 实现 → `Platform.select` of undefined）。
- **解法**：自定义 `resolveRequest` **绝不能覆盖**，必须**委托回** RNOH 的原始实现——保存它、处理完自己的特例后 fallback 调用它：
  ```js
  const harmonyConfig = createHarmonyMetroConfig({ reactNativeHarmonyPackageName: '@react-native-oh/react-native-harmony' });
  const harmonyResolveRequest = harmonyConfig.resolver && harmonyConfig.resolver.resolveRequest;
  const delegateResolve = (ctx, name, platform) =>
    harmonyResolveRequest ? harmonyResolveRequest(ctx, name, platform) : ctx.resolveRequest(ctx, name, platform);
  // resolver.resolveRequest:
  (context, moduleName, platform) => {
    if (platform === 'harmony') { /* 只处理 RNOH 不管的，如 #19 的 stub */ }
    return delegateResolve(context, moduleName, platform); // 其余全委托回 RNOH
  }
  ```
  改 config 后 `bundle-harmony --dev false --config metro.config.harmony.js --reset-cache` 重打。
  > 教训：**不需要手动重定向 code-push**（RNOH 自动做）。最早 `Couldn't provide CodePush` 的真因就是自定义 resolveRequest 覆盖了 RNOH，把 code-push 自动重定向也带没了——手动补一个只是治标，Platform/mmkv/op-sqlite 的重定向仍在丢。
  > 附：三处 turbo module 名仍须一致——JS `TurboModuleRegistry.get('RTNCodePush')`（移植包 `src/NativeCodePush`）、ArkTS `CodePushPackage.createTurboModule`、自包含 C++ `BaseRTNCodePushPackage`。该名由移植包+HAR 固定，封装在 `@itc/hotfix` 内，业务层只见 `hotfix.*`。

### 22. CodePush 接好名字后仍崩：`HostObject::get for prop 'RTNCodePush': Cannot load property of null or undefined`
- **现象**：#21 修好后再跑，错误前进到 turbo module **构造**阶段：`CodePushBuilder.ts:28` / `CodePush.ts:59,64` 报 `Cannot load property of null or undefined`，调用链 `getModule → createTurboModule → CodePushBuilder`。
- **原因**：CodePush 移植包在 native 侧构造时读 `AppStorage.Get("CodePushConfig")[deploymentKeyConfigName]`（还有 `["PublicKey"]`/`["ServerUrl"]`）。若启动时没人往 `AppStorage` 写过 `"CodePushConfig"`，`Get` 返回 undefined，在其上取属性即崩。这是移植包要求开发者预置的运行时配置（编译期 `build-profile.json5` 的 `buildProfileFields` 不会自动进 AppStorage）。
- **解法**：在 [EntryAbility.ets](../apps/oa/harmony/entry/src/main/ets/entryability/EntryAbility.ets) override `onCreate`，**在 `super.onCreate` 前**（早于 RN 实例化与 turbo module 构造）写入：
  ```ts
  interface CodePushConfig { Staging: string; Production: string; ServerUrl: string; PublicKey: string; }
  onCreate(want: Want, launchParam: AbilityConstant.LaunchParam): void {
    AppStorage.setOrCreate<CodePushConfig>('CodePushConfig',
      { Staging: '', Production: '', ServerUrl: '', PublicKey: '' });
    super.onCreate(want, launchParam);
  }
  ```
  - 纯 ArkTS 改动，**不碰 JS bundle**，无需重打 bundle / 清 metro，DevEco 直接 Build & Run。
  - 值留空 = 待接自建 code-push-server；初始化不再崩，`sync`/`checkForUpdate` 因无 key/server 优雅失败（被 `ItcError` 包）。将来填 deployment key 改这里（或改为读 `BuildProfile.*`）。

## 工作习惯

### 18. 后台命令明明失败却报 exit 0
- **原因**：命令以 `| tail` 结尾，捕获到的是 `tail` 的退出码，且日志被截断。
- **解法**：长构建**不要**管道到 tail；让后台原样捕获完整日志，事后 `grep`。

## 设计约束（非 bug，但需牢记）

- **TurboModule spec 入参用基本类型**（string/boolean），不用 object —— 规避 iOS codegen 为对象入参生成 C++ 结构体，三端签名才简单一致。
- **生物绑定公钥格式三端不一致**：Android=DER(SPKI)，iOS/鸿蒙=X9.63（`04||X||Y`）。后端验签需按平台标识归一化。
